# Elasticsearch

**Автор:** Oleborn

**Дата:** 01 марта 2026 г.

**Проект:** [OrderHub](https://github.com/Oleborn/OrderHub)


Elasticsearch — это не просто база данных или поисковый движок; это сложная распределенная система, построенная на базе Apache Lucene, предназначенная для высокоскоростного полнотекстового поиска, аналитики и агрегации данных в реальном времени. В контексте Observability, Elasticsearch является краеугольным камнем для хранения и анализа структурированных логов, метрик и трассировок.

## 1. Фундаментальные концепции

Понимание базовых строительных блоков Elasticsearch критически важно для эффективного использования и масштабирования системы.

### 1.1. Индексы, шарды и реплики

*   **Индекс (Index)**: В Elasticsearch индекс является логическим пространством для хранения данных, аналогичным базе данных в реляционных СУБД. Он представляет собой коллекцию документов, которые имеют схожие характеристики. Каждый индекс имеет уникальное имя, которое используется для обращения к нему при индексации, поиске и обновлении данных. Например, для логов часто используются индексы с датой (`logs-2026.02.26`).

*   **Шард (Shard)**: Индекс в Elasticsearch может быть разделен на один или несколько физических сегментов, называемых шардами. Каждый шард является полностью функциональным и независимым экземпляром Lucene индекса. Шардирование позволяет Elasticsearch:
    *   **Масштабировать данные горизонтально**: Распределять большие объемы данных по нескольким узлам кластера.
    *   **Параллелизовать операции**: Выполнять поисковые запросы и индексацию одновременно на нескольких шардах, что значительно ускоряет работу.
    При создании индекса вы указываете количество первичных шардов (Primary Shards). Это количество нельзя изменить после создания индекса без переиндексации данных.

*   **Реплика (Replica)**: Для обеспечения отказоустойчивости и повышения производительности чтения, каждый первичный шард может иметь одну или несколько реплик. Реплика — это точная копия первичного шарда. Если первичный шард выходит из строя, одна из его реплик автоматически становится первичным шардом. Реплики также могут обрабатывать поисковые запросы, распределяя нагрузку и увеличивая пропускную способность чтения. Количество реплик можно динамически изменять для существующего индекса.

    **Пример**: Индекс с 3 первичными шардами и 1 репликой на каждый шард будет иметь 3 первичных шарда и 3 реплики, что в сумме составляет 6 шардов. Для размещения такого индекса потребуется минимум 2 узла, чтобы реплики не находились на том же узле, что и их первичные шарды.

### 1.2. Узлы (Nodes) и кластер (Cluster)

Кластер Elasticsearch состоит из одного или нескольких узлов, которые работают вместе для хранения и обработки данных. Каждый узел выполняет определенные роли:

*   **Master-eligible Node**: Узел, который может быть выбран в качестве Master-узла. Master-узел отвечает за управление состоянием кластера: создание/удаление индексов, распределение шардов, отслеживание состояния узлов. Для стабильности кластера рекомендуется иметь нечетное количество Master-eligible узлов (например, 3), чтобы избежать проблемы 
Split-Brain. Master-узлы не участвуют в поиске или индексации данных, их задача — поддерживать стабильность кластера.

*   **Data Node**: Узел, который хранит данные (шарды) и выполняет операции, связанные с данными: индексацию, поиск, агрегацию. В больших кластерах Data-узлы могут быть разделены по "горячим" (Hot), "теплым" (Warm) и "холодным" (Cold) данным для оптимизации затрат и производительности.

*   **Ingest Node**: Узел, который может выполнять предварительную обработку документов перед их индексацией. Это позволяет трансформировать, обогащать и нормализовать данные с помощью Ingest Pipelines, разгружая клиентские приложения или Logstash.

*   **Coordinating Node**: Любой узел в кластере может выступать в роли Coordinating Node. Он принимает запросы от клиентов, маршрутизирует их к соответствующим шардам, собирает результаты и возвращает их клиенту. Если узел не имеет других ролей (Master, Data, Ingest), он называется Dedicated Coordinating Node и используется для распределения нагрузки в очень больших кластерах.

### 1.3. Жизненный цикл документа

Понимание того, как документ проходит путь от клиентского запроса до индексации и последующего поиска, является ключом к оптимизации производительности.

#### 1.3.1. Индексация (Write Path)

Когда клиент отправляет запрос на индексацию документа, происходит следующая последовательность событий:

1.  **Маршрутизация (Routing)**: Elasticsearch определяет, на какой первичный шард должен быть отправлен документ. Это делается на основе `_id` документа (если указан) или случайным образом. Формула маршрутизации обычно `shard = hash(routing) % number_of_primary_shards`.

2.  **Запись на первичный шард**: Запрос отправляется на узел, где находится первичный шард. Документ сначала записывается в **буфер памяти (in-memory buffer)** и одновременно в **журнал транзакций (Translog)**. Translog — это Write-Ahead Log (WAL), который обеспечивает надежность: если узел выйдет из строя до того, как данные будут сброшены на диск, они могут быть восстановлены из Translog.

3.  **Репликация**: После успешной записи на первичный шард, документ реплицируется на все реплики этого шарда. Только после подтверждения от первичного шарда и всех активных реплик (или настроенного кворума) операция индексации считается успешной и возвращает ответ клиенту.

4.  **Refresh (Near Real-Time Search)**: Периодически (по умолчанию каждые 1 секунду) содержимое буфера памяти сбрасывается в новый **сегмент Lucene (Lucene Segment)**. Сегмент — это неизменяемая структура данных на диске. После создания сегмента, данные становятся доступными для поиска. Это объясняет "почти в реальном времени" (Near Real-Time, NRT) природу поиска в Elasticsearch: данные доступны для поиска не мгновенно, а с небольшой задержкой.

5.  **Flush**: Периодически (или при достижении определенного размера Translog) происходит операция `flush`. В ходе `flush` все сегменты, созданные с момента последнего `flush`, записываются на диск, Translog очищается, и создается новая точка фиксации (commit point). Это обеспечивает постоянство данных.

#### 1.3.2. Поиск (Read Path)

Поисковый запрос в Elasticsearch — это распределенная операция, которая обычно выполняется в два этапа: Query Phase и Fetch Phase [1].

1.  **Query Phase (Фаза Запроса)**:
    *   **Распространение запроса**: Coordinating Node (узел, принявший запрос) пересылает поисковый запрос на все шарды, которые могут содержать релевантные данные (как первичные, так и реплики).
    *   **Локальный поиск**: Каждый шард выполняет локальный поиск, собирает результаты (например, ID документов, релевантность) и возвращает их Coordinating Node. На этом этапе возвращаются только метаданные, а не полные документы.
    *   **Глобальная сортировка**: Coordinating Node собирает все результаты от шардов, сортирует их по релевантности и определяет, какие документы (и в каком порядке) должны быть возвращены клиенту.

2.  **Fetch Phase (Фаза Извлечения)**:
    *   **Извлечение документов**: Coordinating Node отправляет запросы на те шарды, которые содержат выбранные документы, чтобы получить их полные версии.
    *   **Сборка и возврат**: Получив полные документы, Coordinating Node собирает их, возможно, выполняет дополнительные операции (например, подсветку совпадений) и возвращает финальный результат клиенту.

Этот двухфазный подход позволяет минимизировать сетевой трафик, так как на первом этапе передаются только ID документов, а полные документы извлекаются только те, которые действительно нужны.

## 2. Внутреннее устройство Lucene

Elasticsearch построен на Lucene — высокопроизводительной библиотеке для полнотекстового поиска. Понимание Lucene позволяет глубже понять, как Elasticsearch достигает своей скорости и гибкости.

### 2.1. Инвертированный индекс (Inverted Index)

В основе Lucene лежит концепция инвертированного индекса. В отличие от традиционных баз данных, которые хранят данные построчно и ищут по ним, инвертированный индекс хранит список документов для каждого уникального слова (термина). Это делает полнотекстовый поиск невероятно быстрым.

**Пример**: Допустим, у нас есть два документа:
*   Документ 1: "The quick brown fox jumps over the lazy dog"
*   Документ 2: "Quick brown cats jump high"

Инвертированный индекс будет выглядеть примерно так:

| Термин  | Документы | Позиции |
| :------ | :-------- | :------ |
| brown   | 1, 2      | 3, 2    |
| cats    | 2         | 3       |
| dog     | 1         | 9       |
| fox     | 1         | 4       |
| high    | 2         | 5       |
| jump    | 2         | 4       |
| jumps   | 1         | 5       |
| lazy    | 1         | 8       |
| over    | 1         | 6       |
| quick   | 1, 2      | 2, 1    |
| the     | 1         | 1, 7    |

Когда вы ищете "quick brown", Lucene мгновенно находит оба термина в индексе и пересекает списки документов, чтобы найти те, которые содержат оба слова. Это гораздо быстрее, чем сканировать каждый документ.

### 2.2. Сегменты Lucene (Lucene Segments)

Как упоминалось ранее, данные в Lucene хранятся в неизменяемых сегментах. Каждый сегмент — это полностью независимый, самодостаточный индекс. Он содержит инвертированный индекс, а также другие структуры данных, такие как:

*   **Stored Fields**: Оригинальное содержимое полей документа.
*   **Term Vectors**: Позиции и смещения терминов в документе (используется для подсветки).
*   **Doc Values**: Колоночное хранилище для полей, используемых в сортировке и агрегациях (значительно ускоряет эти операции).
*   **Norms**: Факторы нормализации для оценки релевантности.

**Неизменяемость сегментов** имеет свои преимущества (простота кэширования, отсутствие блокировок при чтении) и недостатки (для обновления документа нужно создать новый сегмент, а старый пометить как удаленный). Это приводит к необходимости периодического слияния сегментов.

### 2.3. Слияние сегментов (Segment Merging)

По мере индексации данных создается множество мелких сегментов. Слишком большое количество сегментов замедляет поиск (так как нужно искать по всем сегментам) и потребляет больше ресурсов. Lucene (и Elasticsearch) автоматически выполняет **слияние сегментов (Segment Merging)**. Этот процесс объединяет несколько мелких сегментов в один большой, удаляя при этом помеченные как удаленные документы.

*   **Merge Policy**: Определяет, когда и какие сегменты следует объединять. Наиболее распространенная политика — `TieredMergePolicy`, которая объединяет сегменты схожего размера.
*   **Оптимизация**: Слияние сегментов — это ресурсоемкая операция (I/O, CPU). Elasticsearch старается выполнять ее в фоновом режиме, чтобы не влиять на производительность индексации и поиска. Однако, при высокой нагрузке на запись, слияние может стать узким местом.

## 3. Маппинги (Mappings) и типы полей

Маппинг — это схема, которая определяет, как Elasticsearch должен хранить и индексировать поля в документах. Он определяет тип данных для каждого поля (строка, число, дата, булево) и то, как это поле будет анализироваться.

### 3.1. Динамический и явный маппинг

*   **Динамический маппинг (Dynamic Mapping)**: По умолчанию Elasticsearch пытается автоматически определить тип поля при первой индексации документа. Это удобно для быстрого старта, но может привести к неоптимальным маппингам (например, числовое поле может быть проиндексировано как текст).

*   **Явный маппинг (Explicit Mapping)**: Рекомендуется для production-систем. Вы явно определяете маппинг для каждого поля, что дает полный контроль над тем, как данные хранятся и индексируются. Это предотвращает ошибки и обеспечивает консистентность.

    ```json
    PUT /my_logs
    {
      "mappings": {
        "properties": {
          "timestamp": { "type": "date" },
          "level": { "type": "keyword" },
          "message": { "type": "text" },
          "traceId": { "type": "keyword" },
          "service.name": { "type": "keyword" }
        }
      }
    }
    ```

### 3.2. Ключевые типы полей для Observability

*   `text`: Для полнотекстового поиска. Значения поля анализируются (разбиваются на слова, приводятся к нижнему регистру и т.д.). Не подходит для агрегаций и точного поиска.

*   `keyword`: Для точного поиска, сортировки и агрегаций. Значения поля индексируются "как есть", без анализа. Идеально подходит для идентификаторов (`traceId`, `userId`), кодов ошибок, уровней логов (`level`).

*   `date`: Для полей, содержащих даты и время. Позволяет выполнять запросы по диапазонам времени и агрегации по временным интервалам.

*   `long`, `integer`, `double`, `float`: Для числовых значений. Используются для агрегаций, сортировки и запросов по диапазонам.

*   `boolean`: Для булевых значений.

**Важно**: Для полей, которые должны быть доступны как для полнотекстового поиска, так и для точного поиска/агрегаций, используйте `multi-fields` (например, `message` как `text` и `message.keyword` как `keyword`).

## 4. Поиск и анализ

Elasticsearch предоставляет мощный язык запросов (Query DSL) и возможности агрегации для извлечения ценной информации из данных.

### 4.1. Query DSL (Domain Specific Language)

Query DSL — это гибкий, основанный на JSON язык для определения поисковых запросов. Он позволяет комбинировать различные типы запросов для достижения высокой точности.

*   **Match Query**: Полнотекстовый поиск по полю.
    ```json
    GET /my_logs/_search
    {
      "query": {
        "match": {
          "message": "order created"
        }
      }
    }
    ```

*   **Term Query**: Точный поиск по полю `keyword`.
    ```json
    GET /my_logs/_search
    {
      "query": {
        "term": {
          "level": "ERROR"
        }
      }
    }
    ```

*   **Range Query**: Поиск по диапазону значений (числа, даты).
    ```json
    GET /my_logs/_search
    {
      "query": {
        "range": {
          "timestamp": {
            "gte": "now-1h",
            "lt": "now"
          }
        }
      }
    }
    ```

*   **Bool Query**: Комбинирование нескольких запросов с логическими операторами (`must`, `filter`, `should`, `must_not`).
    ```json
    GET /my_logs/_search
    {
      "query": {
        "bool": {
          "must": [
            { "term": { "level": "ERROR" } },
            { "match": { "message": "database connection" } }
          ],
          "filter": [
            { "range": { "timestamp": { "gte": "now-1h" } } }
          ]
        }
      }
    }
    ```

### 4.2. Агрегации (Aggregations)

Агрегации позволяют группировать данные и вычислять метрики по этим группам. Это основа для построения дашбордов и аналитики.

*   **Terms Aggregation**: Группировка по уникальным значениям поля (например, количество ошибок по сервисам).
    ```json
    GET /my_logs/_search
    {
      "size": 0,
      "aggs": {
        "errors_by_service": {
          "terms": {
            "field": "service.name.keyword",
            "size": 10
          }
        }
      }
    }
    ```

*   **Date Histogram Aggregation**: Группировка по временным интервалам (например, количество логов в минуту).
    ```json
    GET /my_logs/_search
    {
      "size": 0,
      "aggs": {
        "logs_over_time": {
          "date_histogram": {
            "field": "timestamp",
            "fixed_interval": "1m"
          }
        }
      }
    }
    ```

*   **Metric Aggregations**: Вычисление метрик (среднее, сумма, минимум, максимум) по числовым полям.
    ```json
    GET /my_logs/_search
    {
      "size": 0,
      "aggs": {
        "avg_duration": {
          "avg": {
            "field": "duration_ms"
          }
        }
      }
    }
    ```

## 5. Управление жизненным циклом индексов (ILM)

Логи — это данные, которые постоянно растут и со временем теряют свою актуальность. Index Lifecycle Management (ILM) позволяет автоматизировать управление индексами, оптимизируя затраты и производительность.

### 5.1. Фазы жизненного цикла

ILM определяет политику, которая применяется к индексу на протяжении его жизни, проходя через различные фазы:

*   **Hot Phase**: Индекс активно записывается и часто запрашивается. На этом этапе данные хранятся на быстрых дисках.
    *   Действия: `rollover` (создание нового индекса при достижении порога размера или возраста), `set_priority`.

*   **Warm Phase**: Индекс больше не записывается, но все еще часто запрашивается. Данные могут быть сжаты или перенесены на более медленные диски.
    *   Действия: `shrink` (уменьшение количества первичных шардов), `force_merge` (принудительное слияние сегментов для оптимизации поиска), `allocate` (перемещение на другие узлы).

*   **Cold Phase**: Индекс редко запрашивается. Данные переносятся на самые дешевые диски (например, HDD) или в облачное хранилище.
    *   Действия: `freeze` (освобождение памяти путем выгрузки данных из кэша), `allocate`.

*   **Delete Phase**: Индекс больше не нужен и удаляется.
    *   Действия: `delete`.

### 5.2. Пример политики ILM для логов

```json
PUT /_ilm/policy/my_log_policy
{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": {
            "max_age": "7d",
            "max_size": "50gb"
          }
        },
        "min_age": "0ms"
      },
      "warm": {
        "min_age": "14d",
        "actions": {
          "forcemerge": {
            "max_num_segments": 1
          },
          "shrink": {
            "number_of_shards": 1
          }
        }
      },
      "cold": {
        "min_age": "30d",
        "actions": {
          "freeze": {}
        }
      },
      "delete": {
        "min_age": "90d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}
```

Эта политика будет создавать новый индекс каждые 7 дней или при достижении 50 ГБ, через 14 дней оптимизировать его, через 30 дней замораживать и через 90 дней удалять.

## 6. Производительность и оптимизация

Тонкая настройка Elasticsearch требует глубокого понимания его внутренних механизмов и особенностей рабочей нагрузки.

### 6.1. Настройка JVM и системные лимиты

*   **Heap Size (Размер Кучи)**: Один из самых критичных параметров. Установите `ES_JAVA_OPTS` в `jvm.options`. Рекомендуется выделять не более 50% доступной оперативной памяти (но не более 32 ГБ), оставляя остальное для кэша файловой системы ОС. Это позволяет Lucene эффективно использовать кэш для сегментов.

*   **Swapping**: Полностью отключите свопинг (`vm.swappiness = 1` или `bootstrap.memory_lock: true`). Свопинг JVM на диск приводит к катастрофическому падению производительности.

*   **File Descriptors**: Увеличьте лимиты на количество открытых файловых дескрипторов (`ulimit -n`). Elasticsearch активно работает с файлами Lucene.

*   **Thread Limits**: Увеличьте лимиты на количество потоков (`ulimit -u`).

### 6.2. Оптимизация индексации

*   **Bulk API**: Всегда используйте Bulk API для индексации нескольких документов. Это значительно снижает накладные расходы на HTTP-запросы и сетевой трафик.

*   **Refresh Interval**: Для высоконагруженных систем логирования, где NRT-поиск не критичен каждую секунду, можно увеличить `index.refresh_interval` до 30-60 секунд. Это уменьшит количество создаваемых сегментов и нагрузку на слияние.

*   **Number of Replicas**: Временно уменьшите количество реплик до 0 во время массовой загрузки данных, а затем верните их обратно. Это ускорит индексацию, так как не нужно ждать репликации.

*   **Disable `_source`**: Если вам не нужно хранить оригинальный JSON-документ (например, если вы индексируете только для поиска и агрегаций), отключите `_source`. Это сэкономит место на диске.

### 6.3. Оптимизация поиска

*   **Doc Values**: Убедитесь, что поля, используемые для сортировки и агрегаций, имеют тип `keyword` или числовой. Elasticsearch использует Doc Values для этих операций, что очень эффективно.

*   **Fielddata**: Избегайте использования `fielddata: true` для `text` полей. Это очень ресурсоемкая операция, которая может привести к OutOfMemoryError. Если нужно агрегировать по `text` полю, используйте `multi-fields` с `keyword`.

*   **Cache**: Elasticsearch активно использует кэши (Fielddata Cache, Request Cache). Оптимизируйте запросы, чтобы они могли эффективно использовать эти кэши.

*   **Shard Count**: Оптимальное количество шардов — это баланс между параллелизмом и накладными расходами. Слишком много шардов на узел приводит к увеличению потребления памяти и CPU. Рекомендуется 1-3 шарда на ГБ RAM.

## 7. Мониторинг и устранение неполадок

Эффективный мониторинг кластера Elasticsearch критически важен для его стабильной работы.

### 7.1. Ключевые метрики

*   **JVM Heap Usage**: Отслеживайте использование кучи JVM. Высокое использование может указывать на проблемы с запросами или утечки памяти.
*   **CPU Usage**: Высокое использование CPU может быть вызвано тяжелыми запросами, интенсивной индексацией или слиянием сегментов.
*   **Disk I/O**: Мониторинг операций чтения/записи на диск. Высокая активность может указывать на проблемы с производительностью дисковой подсистемы.
*   **Indexing Rate**: Скорость индексации документов.
*   **Search Latency**: Задержка поисковых запросов.
*   **Segment Count/Size**: Отслеживайте количество и размер сегментов. Слишком много мелких сегментов — признак проблем со слиянием.
*   **Rejected Threads**: Количество отклоненных запросов в пулах потоков (например, `bulk`, `search`). Указывает на перегрузку кластера.

### 7.2. Инструменты мониторинга

*   **Elastic Stack Monitoring**: Встроенный инструмент в Kibana для мониторинга кластера Elasticsearch.
*   **Prometheus + Grafana**: Можно собирать метрики Elasticsearch через JMX Exporter или встроенный `/_nodes/stats` API и визуализировать в Grafana.

### 7.3. Распространенные проблемы и решения

*   **Cluster Red/Yellow**: Красный статус означает потерю первичных шардов (данные недоступны). Желтый — потеря реплик (данные доступны, но нет отказоустойчивости). Решение: восстановить узлы, добавить новые узлы, перераспределить шарды.
*   **OutOfMemoryError**: Чаще всего из-за неправильной настройки Heap Size или тяжелых запросов/агрегаций на `text` полях без `keyword`.
*   **Медленный поиск**: Может быть вызван большим количеством мелких сегментов, неоптимизированными запросами, недостатком памяти для кэша или медленной дисковой подсистемой.

## 8. Безопасность

Безопасность в Elasticsearch критически важна, особенно при работе с чувствительными данными.

### 8.1. X-Pack Security

Elasticsearch предоставляет X-Pack Security (ранее платная, теперь доступна в базовой версии) для обеспечения безопасности:

*   **Аутентификация**: Пользователи, роли, интеграция с LDAP/Active Directory.
*   **Авторизация**: Управление доступом на уровне индексов, документов и полей.
*   **Шифрование**: HTTPS для REST API, шифрование трафика между узлами.

### 8.2. Сетевая безопасность

*   **Firewall**: Ограничьте доступ к портам Elasticsearch (9200 для HTTP, 9300 для межкластерного взаимодействия) только для доверенных IP-адресов.
*   **VPN/VPC**: Размещайте кластер в приватной сети.

## 9. Масштабирование

Elasticsearch изначально разработан для горизонтального масштабирования.

### 9.1. Горизонтальное масштабирование

*   **Добавление узлов**: Самый простой способ увеличить емкость и производительность. Elasticsearch автоматически перераспределит шарды по новым узлам.
*   **Разделение ролей**: В больших кластерах рекомендуется разделять узлы по ролям (Master-only, Data-only, Ingest-only) для оптимизации ресурсов.

### 9.2. Облачные решения

*   **Elastic Cloud**: Управляемый сервис Elasticsearch от Elastic. Упрощает развертывание, масштабирование и обслуживание.
*   **AWS OpenSearch Service, Azure Elasticsearch, Google Cloud Elasticsearch**: Управляемые сервисы от облачных провайдеров.

## 10. Заключение

Elasticsearch — это мощный и сложный инструмент, который требует глубокого понимания для эффективного использования. Он является незаменимым компонентом в стеке Observability, позволяя инженерам быстро находить, анализировать и устранять проблемы в распределенных системах. Освоение его архитектуры, механизмов индексации и поиска, а также методов оптимизации, открывает широкие возможности для создания высокопроизводительных и отказоустойчивых решений.
